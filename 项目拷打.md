### 第一轮：架构与核心设计拷打

**Q1. 关于 AgentCore 的“大脑”职责：**
你提到 `AgentCore` 是核心编排器，负责意图识别、RAG 注入、记忆管理。但我看到你在 `chat()` 方法里，是先做意图识别，如果是 RAG 意图就直接把检索结果拼接到 UserMessage 里。

*   **拷打点：** 这种“前置 RAG”和 Spring AI 原生的 `Tool Calling`（Function Calling）有什么本质区别？如果我把 `knowledge_search` 注册为一个 Tool，让 LLM 自己决定要不要查知识库，和你现在这种写死在代码里的 `if (intent == Intent.RAG)` 相比，优劣在哪里？你这样写是不是剥夺了 LLM 的自主决策权？
*   回答：
    *   **本质区别在于“确定性”与“灵活性”。 前置 RAG 是硬编码逻辑（Hard-coded），只要意图识别说是 RAG，就必查；而 Tool Calling 是概率决策，LLM 可能会因为上下文太长、温度太高或者模型本身变傻而“忘记”调用工具。**
    *   **你没提到的关键点： 前置 RAG 的优势是减少 ReAct 循环的次数。如果走 Tool Calling，LLM 需要先输出 JSON -> Agent 执行 -> 结果返回 LLM -> LLM 再输出最终答案。这至少需要 2-3 次 API 交互。前置 RAG 在一次 Prompt 里就把背景知识给到了，效率在某些场景下反而更高。**
    *   **改进建议： 现在的趋势是 Hybrid（混合模式）：先用轻量级意图识别，如果是强知识相关，直接前置注入；如果是模糊查询，交给 Tool Calling 让 LLM 自己去探索。**


**Q2. 意图识别的延迟与成本：**
你使用 LLM 来做意图识别（`IntentRecognizer`）。

*   **拷打点：** 这意味着用户每发一句话，你都要先调一次 LLM 做分类，再调一次 LLM 做正式回复。这直接导致**首字延迟（TTFT）翻倍**，API 成本也翻倍。你有没有考虑过用轻量级的本地模型（如 ONNX 跑一个小的文本分类模型）或者简单的关键词匹配来替代？如果我现在要求你把意图识别的延迟降到 10ms 以内，你该怎么改？
*   回答：
    *   10ms 怎么做到？ 根本不需要 LLM！
    *   关键词匹配（Trie 树/AC 自动机）： 比如检测到“Java”、“Spring”、“配置”等关键词，直接打标 RAG。这在 Java 里跑起来是微秒级的。
    *   本地小模型（ONNX/BERT）： 跑一个 bert-base-multilingual 做文本分类，在 CPU 上也就 10-50ms。
    *   向量相似度： 预存几个典型的 RAG 问题向量，用户输入来了算个余弦相似度，超过阈值就认为是 RAG。
        结论： 意图识别这种简单任务，用 LLM 属于“高射炮打蚊子”，既贵又慢。


**Q3. ChatMemory 的线程安全问题：**
你在 `AgentCore` 里用了 `ConcurrentHashMap<String, ChatMemory>` 来隔离会话。

*   **拷打点：** 虽然 Map 是线程安全的，但 `ChatMemory` 内部呢？我看你的 `getMessages()` 方法里会触发 `compressIfNeeded()`，这会修改 `history` 列表并调用 LLM。如果同一个 `sessionId` 有两个并发请求进来，同时触发压缩，会不会导致 `history` 被重复清空或者摘要错乱？你做过并发测试吗？
*   回答：同一个 ChatMemory 实例在多线程环境下是不安全的，因为多个线程共享同一个 history（ArrayList数组） 对象，且操作不是原子的，会导致数据竞争和状态不一致。需要在关键方法上加锁，保证操作的原子性。可以使用synchronized锁或者ReentrantLock。

### 第二轮：RAG 模块深度拷打

**Q4. 向量存储的性能瓶颈：**
你的 `VectorStore` 是一个内存实现（看 README 提到的）。

*   **拷打点：** 如果我的知识库从几 KB 变成几个 GB，有几百万个 Chunk，你的内存向量存储会发生什么？OOM（内存溢出）吗？另外，内存存储重启后数据就丢了，你是怎么解决持久化问题的？如果让你迁移到 Milvus 或 Elasticsearch，你需要改动多少代码？你的 `Retriever` 接口设计支持这种无缝切换吗？
*   回答：本地内存存不了太大的向量库，否则内存直接满了。我们可以把内存向量库改成数据库向量库，来实现持久化存储。最好的方式就是先实现一个VectorStore 接口，包含 add(Document) 和 search(Vector, int k) 方法等通用方法。

**Q5. 多路召回的 RRF 融合：**
你用了 RRF（倒数排名融合）来合并语义检索和 BM25 的结果。

*   **拷打点：** RRF 的公式是 $1/(k+rank)$，你默认 $k=60$。这个 60 是怎么来的？有没有针对你的业务数据做过调优？如果语义检索排第一，BM25 排第十，RRF 可能会偏向谁？在你的场景下，RRF 真的比加权分数融合（Weighted Sum）好吗？
*   回答：
    *   为什么是 60？ 这是一个经验值。在信息检索领域（如 TREC 竞赛），研究发现 $k=60$ 能在“排名靠前的文档”和“排名靠后的文档”之间取得最好的平衡。
        	如果 $k$ 太小（比如 1）：第一名得分 $1/2$，第二名 $1/3$，差距巨大，容错率低。
        	如果 $k$ 太大（比如 1000）：所有分数的差距都被抹平了，排序失去了意义。
    *   偏向谁？ RRF 是无偏的。它只看排名。如果语义检索排第一（得分 $1/61$），BM25 排第十（得分 $1/70$），最后融合时，这个文档会获得两者的分数之和。RRF 的核心价值在于：它不需要关心不同算法给出的原始分数（Score）量级是否一致（语义相似度可能是 0.9，BM25 可能是 15.2，没法直接加）。
    *   怎么调优？ 准备一批“问题-标准答案”对，调整 $k$ 值，看哪种 $k$ 能让标准答案排在最前面（计算 MRR 或 NDCG 指标）。


**Q6. Rerank 模型的依赖：**
你的 `LlmReranker` 是直接调用的智谱 API。

*   **拷打点：** 这是一个同步的 HTTP 调用。如果智谱的 Rerank 接口响应慢了（比如 2s），你的整个 RAG 流程就会卡住。你有没有做异步处理或者超时降级？如果 Rerank 失败了，你直接返回原始召回结果，这会不会导致不相关的文档被送入 LLM，产生幻觉？
*   回答：
    *   超时控制： 必须在 HTTP Client 层面设置 timeout。比如设置 3 秒超时，一旦超过，立刻抛出异常。
    *   降级策略（Fallback）： 当 Rerank 失败或超时时，绝对不能让整个流程崩掉。你应该捕获异常，然后直接使用多路召回的原始结果（或者只取前 3 个）。虽然精度差一点，但至少能保证服务可用（Availability）。
    *   异步优化： 如果 Rerank 真的很慢，可以考虑在召回阶段就启动异步线程去跑 Rerank，或者使用本地部署的 Rerank 模型（如 BGE-Reranker），通过 ONNX Runtime 在本地 CPU/GPU 上跑，延迟能控制在 100ms 以内。
        *   如果你的 RAG 系统有多路召回（比如同时跑语义检索、BM25、关键词检索）：
            并行执行： 你可以让这三路检索同时在不同的线程里跑。
            异步 Rerank： 当其中任意一路（比如最快的 BM25）先拿到了结果，就可以先把这部分结果丢给 Rerank 模型去预热或初步排序，而不必等最慢的那路（比如向量检索）全部完成。
            最终融合： 等所有路都回来后，再做最终的 RRF 融合和二次 Rerank。
